Lab 1: Preliminary lab for practicals

UCD server and clusters to use and experiment:

1) csserver.ucd.ie  - 2 cores machine, you will use it to understand cache blocking and matrix multiplication

2) yeats.ucd.ie       - 16 cores machine, you will use it for Pthreads and OpenMP examples

3) csicluster.ucd.ie - 6 node distributed memory cluster to use for MPI experiments

You will use one username for all the machines.

The first step is to check if you can SSH access to the servers:

ssh <username>@csserver.ucd.ie

Linux/Mac users should have SSH clients. For Windows, one possible SSH client is Putty.

On Max there may be some problem with ssh X11 forwarding. Please have a look to the link below: http://www.cyberciti.biz/faq/apple-osx-mountain-lion-mavericks-install-xquartz-server/

Setting up MPI environment

1. Log into the csicluster:
ssh <username>@csicluster.ucd.ie

2. Create a machinefile in your preferred directory with the names of the nodes of csicluster:

  csicluster01
 #csicluster02

Here I commented out csicluster02 because it is not working at the moment.

You can create this file yourself or copy to your home dir:

cp /home/cs/khasanov/COMP40730/hostfile .

3.There is a number of MPI installations on csicluster. To use the easiest one (Open MPI), do:
export PATH=/usr/bin:$PATH

You can add the export statement in the .bashrc file in your home directory to make it permanent.


4. Create a hello world MPI program in C or C++ (or in Fortran  :) ) to see if everything works properly:

  #include <stdio.h>
  #include <stdlib.h>
  #include <mpi.h>

 int main( int argc, char** argv )
 {
    int myId, numProc, len;
    char name[MPI_MAX_PROCESSOR_NAME];
    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &myId);
    MPI_Comm_size(MPI_COMM_WORLD, &numProc);
    MPI_Get_processor_name(name, &len);
    printf("Id: %d, Name: %s, NumProcs: %d\n", myId, name, numProc);
    MPI_Finalize();
    return 0;
 }

You can create this file yourself or copy it to your home as follows:

cp /home/cs/khasanov/COMP40730/HelloWorldMPI.c .



5. Compile using:
mpicc -o <binary> <C source>
or
mpic++ -o <binary> <C++ source>

6. MPI uses SSH to launch processes on different nodes. There is a script which will make your life much easier when setting up SSH. Create a shell script named fix-ssh.sh with the following content:

ssh-keygen -N "" -f "$HOME/.ssh/id_rsa"
cat $HOME/.ssh/id_rsa.pub >> $HOME/.ssh/authorized_keys
ssh-keyscan csicluster01 >> $HOME/.ssh/known_hosts
#ssh-keyscan csicluster02 >> $HOME/.ssh/known_hosts
ssh-keyscan csicluster03 >> $HOME/.ssh/known_hosts
ssh-keyscan csicluster04 >> $HOME/.ssh/known_hosts
ssh-keyscan csicluster05 >> $HOME/.ssh/known_hosts
ssh-keyscan csicluster06 >> $HOME/.ssh/known_hosts

You can create this file yourself or copy to your home dir:

cp /home/cs/khasanov/COMP40730/fix-ssh.sh .

Then source the created script as follows:

source ./fix-ssh.sh

7. Run the programs for example with 5 processes (1 process per node) using:
mpirun -np 6 --hostfile hostfile <binary>

Cache Blocking

Cache blocking attemps to decrease the cache-miss rates by enhancing the temporal locality and spatial locality concepts. Temporal locality implies that a data item used now is likely to be used again soon. Spatial locality implies that if a data item is referenced at a particular time, then it is likely that neighbouring locations will be referenced in the near future. A loop computation such as matrix multiplication, can lead to frequent cache misses if the array size exceeds the cache size. In cache blocking, the loops can be restructured so that the computation proceeds in contiguous blocks chosen to fit in the cache. If the block size is chosen properly, then the overall time needed for the computation can be decreased significantly. More information on cache blocking can be found here.
It is recommended to have a look at least Section 3 in the article named "What Every Programmer Should Know About Memory "  on the link http://people.freebsd.org/~lstewart/articles/cpumemory.pdf .

for(bi=0; bi<n; bi+=blockSize)

for(bj=0; bj<n; bj+=blockSize)

for(bk=0; bk<n; bk+=blockSize)

for(i=0; i<blockSize; i++)

for(j=0; j<blockSize; j++)

for(k=0; k<BlockSize; k++)

C[bi+i][bj+j] += A[bi+i][bk+k]*B[bk+k][bj+j]

NOTE: the block size must be a factor of the matrix size!

Matrix Multiplication with BLAS/ATLAS

BLAS (Basic Linear Algebra Subprograms) are routines that provide standard building blocks for performing basic vector and matrix operations.

- Because the BLAS are efficient, portable, and widely available, they are commonly used in the development of high quality linear algebra software, LAPACK for example.

- These subroutines are very carefully thought out programs that exploit several techniques including Loop-Unrolling and Cache Blocking.

There are three levels:

Level 1 BLAS perform scalar, vector and vector-vector operations

Level 2 BLAS perform matrix-vector operations

Level 3 BLAS perform matrix-matrix operations.

CBLAS: C interface for BLAS

BLAS subroutine DGEMM (Double precision GEneral Matrix Multiply)

This function calculates C = x*A*B + y*C

cblas_dgemm(CblasRowMajor, CblasNoTrans, CblasNoTrans, N, N, N, 1.0, A, N, B, N, 0.0, C, N);

ATLAS

ATLAS stands for Automatically Tuned Linear Algebra Subprograms. ATLAS is based on BLAS but when you install ATLAS, many tests are performed in order to determine the best pararmeters to use for your system. This can give better performance than BLAS because BLAS uses parameters that are designed to maximize portability, not performance on a particular architecture.

Some commercial packages that use ATLAS are: MAPLE(v7 and higher), MATLAB(v6.0 and higher), Mathematica(forthcoming), and Octave.

The program ATLAS.c performs the same function call as BLAS.c, but is faster. The only difference between the two programs is how they are compiled. ATLAS.c is linked to files that contain the parameters that ATLAS has determined to be optimal on the machine it is installed on. The compile commands for ATLAS.c is below.
